## 第二部分：新发现的问题

### 一、严重问题（阻塞开发）

#### 【严重】问题1：歌单封面拼接图的路径管理机制缺失

**位置**：2.2 数据模型 - Playlist、3.3.1 创建歌单

**问题描述**：
文档说明：
- "默认：2x2网格拼接前4首歌封面"
- "缓存生成的封面，避免重复生成"
- "歌曲变化时自动重新生成"
- Playlist模型中：`var customCoverPath: String?`（用户自定义封面）

**核心问题**：
1. **自动生成的拼接封面存储在哪里？**
   - Playlist模型中没有字段存储自动拼接封面的路径
   - 只有customCoverPath（用户自定义）
   - 自动生成的封面路径如何管理？

2. **拼接封面何时重新生成？**
   - "歌曲变化时"是指：歌单添加歌曲？删除歌曲？重新排序？
   - 如果前4首变化，立即重新生成（可能频繁）？
   - 如果用户删除了封面图片文件，如何处理？

3. **缓存清理策略不明确**
   - 拼接封面缓存在哪个目录？
   - 应用退出或清理缓存时，拼接封面会被删除吗？
   - 如果被删除，下次如何重新生成？

**影响**：
- 开发人员不知道如何实现自动拼接封面功能
- 可能导致数据不一致或性能问题
- 用户可能遇到封面显示异常

**建议**：
**方案A**（推荐）：在Playlist模型中增加字段
```swift
struct Playlist {
    // ... 现有字段
    var autoCoverPath: String?      // v4.4新增：自动生成的拼接封面路径
    var autoCoverHash: String?      // v4.4新增：前4首歌曲的hash（用于判断是否需要重新生成）
    var customCoverPath: String?    // 用户自定义封面（优先级高于autoCoverPath）
}
```

**封面逻辑**：
1. 显示封面时：如果customCoverPath不为nil，使用自定义封面；否则使用autoCoverPath
2. autoCoverPath为nil或autoCoverHash与当前前4首不匹配时，重新生成
3. 生成后保存到Caches/PlaylistCovers/目录，路径存入autoCoverPath
4. 清理缓存时，可以删除autoCoverPath，但不删除customCoverPath

**方案B**：动态生成，不缓存
- 每次显示时实时生成（性能差）
- 或只在内存中缓存，应用退出后清空
- **不推荐**：可能导致列表滚动卡顿

#### 【严重】问题2：歌词延迟加载机制未完整定义

**位置**：2.2 数据模型 - Song、3.12 歌词显示

**问题描述**：
文档说明：
- "lyrics 从 Song 模型移除，在歌词显示页按需从文件或ID3标签延迟加载"
- 歌词支持优先级：
  1. 内嵌歌词（ID3v2 LYRIC/USLT）
  2. 外置LRC文件（与音乐文件同名的.lrc文件）
  3. 用户手动添加歌词（P3）

**核心问题**：
1. **外置LRC文件存储在哪里？**
   - 是用户手动导入的.lrc文件？
   - 还是应用自动下载的？
   - 如果是手动导入，导入到哪个目录？（Documents/Music/与歌曲同目录？）
   - 如果是自动下载，v4.3明确说"在线歌词匹配"是v2.0功能

2. **用户手动添加歌词后存储在哪里？**
   - 是否保存为独立的.lrc文件？
   - 还是存储在数据库中？
   - 如果是文件，路径如何管理？
   - 是否需要在Song模型中增加lyricsPath字段？

3. **歌词显示页如何"延迟加载"？**
   - 打开歌词页 → 检查ID3标签 → 检查.lrc文件 → 显示默认
   - 这个流程的优先级和性能如何保证？
   - 如果ID3解析很慢，是否会阻塞UI？

4. **编辑歌词（P3）后如何保存？**
   - 是否更新到ID3标签（需要修改原始文件）？
   - 还是保存为独立的.lrc文件？
   - 还是存储在数据库？
   - 文档3.1.10说"编辑仅修改应用内数据库记录"，但歌词不是Song模型的字段

**影响**：
- 歌词功能的核心逻辑不明确
- 开发人员无法实现
- 可能导致数据丢失或性能问题

**建议**：
**方案1**：歌词存储在独立文件
```swift
struct Song {
    // ... 现有字段
    var lyricsPath: String?       // v4.4新增：外置歌词文件路径（相对Documents目录）
    var lyricsSource: LyricsSource? // v4.4新增：歌词来源（embedded/file/manual）
}

enum LyricsSource {
    case embedded    // 内嵌在ID3标签
    case file        // 外置.lrc文件
    case manual      // 用户手动添加（存储在文件）
}
```

**歌词加载逻辑**：
1. 打开歌词页 → 异步加载（不阻塞UI）
2. 优先级：内嵌 > 外置文件 > 手动
3. 用户手动添加的歌词 → 保存到Documents/Lyrics/目录，路径存入lyricsPath
4. P3编辑歌词 → 保存到lyricsPath指向的文件（不修改原歌曲文件）

**方案2**：歌词存储在数据库
- 新增Lyric表，存储songId、lyrics文本、lyricsSource
- 优点：查询快，易于管理
- 缺点：数据库膨胀，同步歌词需要额外处理

#### 【重要】问题4：WiFi传输自定义密码安全性不足

**位置**：3.4.2 WiFi传输 - 安全机制

**问题描述**：
- "可选：允许用户设置自定义密码（至少6位，仅数字）"
- "连续输错密码5次后，冷却30秒"

**核心问题**：
1. **自定义密码仍然是纯数字，容易被暴力破解**
   - 6位数字 = 1,000,000种组合
   - 在局域网内，使用多个线程并发请求，可以在数小时内破解

2. **HTTP传输明文密码**
   - 即使有防暴力破解，密码在HTTP传输过程中仍是明文
   - 如果WiFi网络被监听（如ARP欺骗），密码会被窃取
   - 一旦密码被窃取，防暴力破解机制失效

3. **自定义密码存储在哪里？**
   - 明文存储在UserDefaults？（不安全）
   - 加密存储？（使用什么算法？Keychain？）
   - 文档未说明

**影响**：
- WiFi传输功能存在安全风险
- 可能导致用户的文件被未授权访问
- 在公共WiFi环境中尤其危险

**建议**：
**方案A**（推荐）：移除自定义密码功能
- 只使用随机生成的6位密码
- 每次启动WiFi传输时重新生成
- 密码有效期：当前会话
- 优点：简单、安全、够用

**方案B**：增强自定义密码安全性
- 密码长度提升到8-12位
- 允许字母+数字（组合数：62^12 ≈ 3×10^21）
- 密码使用HTTPS传输（但v1.0不实现HTTPS）
- 密码存储在Keychain中
- **不推荐**：实现复杂，仍不如HTTPS安全

**方案C**：添加安全警告
- 在设置中显示警告："自定义密码在HTTP传输中可能被窃取，建议仅在可信WiFi网络中使用WiFi传输功能"
- 提供默认选项：仅使用随机密码（推荐）/ 自定义密码（高级用户）

**文档说明**：
v4.3文档已说明"仅在可信网络中使用"，但建议：
1. 在WiFi传输页面顶部显示明显的安全警告
2. 自定义密码设置前显示风险提示
3. 考虑移除自定义密码功能（v1.0）

#### 【重要】问题5：小容量设备缓存策略可能导致体验下降

**位置**：3.8.1 图片缓存策略

**问题描述**：
- "设备可用存储 < 1GB 时：自动降低缓存上限至50MB"
- **问题**：
  1. 50MB缓存对于大量音乐用户可能不足
     - 假设每张封面50KB，50MB只能缓存1000张
     - 用户有3000首歌，缓存命中率只有33%
     - 频繁重新加载封面，影响性能和流量（如果从网络获取）

  2. 小容量设备用户（64GB）更容易遇到存储不足
     - 这类用户更需要缓存优化（因为设备性能通常较差）
     - 但v4.3策略反而减少了他们的缓存

**影响**：
- 小容量设备用户体验下降
- 列表滚动可能卡顿（频繁加载图片）
- 电量消耗增加（频繁读取文件）

**建议**：
**方案A**（推荐）：缓存大小与歌曲数量挂钩
```swift
// 计算缓存上限
let songCount = getAllSongs().count
let baseCacheSize = 50 * 1024 * 1024  // 50MB基础值
let dynamicSize = min(
    baseCacheSize + songCount * 50 * 1024,  // 每首歌增加50KB
    300 * 1024 * 1024  // 上限300MB
)
```

**方案B**：提供用户可配置选项
- 设置中提供：缓存大小（小50MB / 中100MB / 大200MB）
- 默认根据设备存储空间自动选择
- 用户可手动调整

**方案C**：优化图片加载策略
- 列表只加载缩略图（100x100px，约5KB）
- 详情页才加载原图（1024x1024px，约50KB）
- 这样50MB可以缓存10000张缩略图

#### 【重要】问题6：批量操作在"全部音乐"视图中的语义不完整

**位置**：3.1.9 批量操作

**问题描述**：
文档说明：
- "在全部音乐：按钮文案为`从音乐库删除`，需二次确认"

**核心问题**：
1. **批量删除的交互流程不明确**
   - 用户选择10首歌 → 点击"从音乐库删除"
   - 是否对每首歌都确认？（繁琐）
   - 还是一次性确认所有歌曲？（可能误删大量歌曲）

2. **是否需要"从歌单移除"选项？**
   - 在"全部音乐"中批量选择，用户可能想：
     - 从某个歌单移除（而不是删除）
     - 从所有歌单移除（但不删除文件）
   - 但当前只提供"从音乐库删除"

3. **与单首歌曲删除的不一致性**
   - 单首左滑：提供"从歌单移除"和"从音乐库删除"两个选项
   - 批量操作：只提供"从音乐库删除"
   - 功能不一致，用户困惑

**影响**：
- 用户无法批量从歌单移除歌曲
- 可能误删大量歌曲
- 交互逻辑不一致

**建议**：
**方案A**（推荐）：提供两种批量删除选项
```
批量选择10首 → 点击[🗑️] → 弹出选择：
  ○ 从全部音乐移除（仅移出"全部音乐"列表，不删除文件）
  ○ 从所有歌单移除（从所有自定义歌单移除，不删除文件）
  ● 从音乐库删除（删除文件，需二次确认）
```

**方案B**：简化为确认对话框
```
批量选择10首 → 点击"从音乐库删除"
→ 弹出确认：
  "确定要从音乐库删除这10首歌曲吗？文件将被永久删除。"
  [取消] [删除]
```

**方案C**：分步操作
1. 批量选择 → 点击"批量操作"
2. 显示操作列表：
   - 批量播放
   - 添加到歌单
   - 添加到我喜欢
   - 从歌单移除
   - 从音乐库删除
3. 用户选择具体操作

**推荐**：采用方案A，提供灵活的批量操作

#### 【重要】问题7：搜索范围定义模糊

**位置**：3.1.2 搜索功能

**问题描述**：
- "搜索歌曲、艺术家、专辑"
- **问题**：搜索范围是什么？
  1. 只搜索当前Tab？（如"艺术家"Tab只搜索艺术家名称）
  2. 搜索全部音乐？
  3. 搜索艺术家+专辑+歌曲名称？

**场景分析**：
- 用户在"艺术家"Tab中输入"周杰伦"
  - 期望：显示所有周杰伦的歌曲
  - 还是：只显示艺术家名称匹配的艺术家卡片？

- 用户在"专辑"Tab中输入"七里香"
  - 期望：显示《七里香》专辑
  - 还是：显示所有包含"七里香"的歌曲？

**影响**：
- 用户期望与实际行为不符
- 搜索功能体验不佳
- 开发实现不明确

**建议**：
明确搜索范围：

**方案A**（推荐）：全局搜索，当前Tab过滤
- 搜索框始终搜索全部音乐库
- 但当前Tab决定结果的显示方式：
  - "全部音乐"Tab：显示歌曲列表
  - "艺术家"Tab：显示艺术家分组（搜索结果按艺术家聚合）
  - "专辑"Tab：显示专辑分组（搜索结果按专辑聚合）
- 在搜索框下方显示提示："搜索到X首歌曲，Y个艺术家，Z个专辑"

**方案B**：当前Tab限制搜索
- 在"全部音乐"Tab：搜索歌曲名、艺术家、专辑
- 在"艺术家"Tab：只搜索艺术家名
- 在"专辑"Tab：只搜索专辑名
- **缺点**：用户可能不知道切换Tab来搜索不同内容

**推荐**：采用方案A，提供更灵活的搜索体验

### 三、一般问题（建议优化）

#### 【一般】问题8：FLAC筛选项显示时机不明确

**位置**：3.1.4 筛选功能

**问题描述**：
- "FLAC/ALAC（无损）— v4.3说明：FLAC解码器未集成前，该筛选项标注'暂不支持播放'"

**核心问题**：
1. **筛选项何时标注"暂不支持播放"？**
   - 是筛选列表中显示？
   - 还是选择该筛选后，搜索结果中标注？
   - 还是筛选按钮上显示？

2. **用户选择"FLAC"筛选后会发生什么？**
   - 显示所有FLAC文件，但标注"暂不支持播放"？
   - 还是弹窗提示"FLAC格式暂不支持，是否切换筛选条件"？
   - 还是直接隐藏FLAC文件（用户可能以为没有FLAC文件）？

3. **ALAC与FLAC的处理不一致**
   - ALAC是P0原生支持
   - FLAC是P2需要第三方解码器
   - 但筛选项是"FLAC/ALAC（无损）"，如何区分？

**影响**：
- 用户可能困惑为什么FLAC文件无法播放
- 交互逻辑不明确

**建议**：
**方案A**（推荐）：拆分筛选项
```
按文件格式筛选：
  - 全部格式
  - MP3
  - M4A/AAC
  - ALAC（无损）
  - FLAC（无损，v1.0暂不支持）
  - WAV
  - 其他
```

选择"FLAC"后：
- 如果库中有FLAC文件：显示提示"FLAC格式将在v1.1版本支持播放"
- 如果库中无FLAC文件：显示"暂无FLAC文件"

**方案B**：隐藏FLAC筛选项
- v1.0不显示"FLAC/ALAC"筛选项
- 在设置中说明"v1.1将支持FLAC格式"
- v1.1集成FLAC解码器后，显示该筛选项

#### 【一般】问题9：最近播放分页加载与"全部"时间范围的冲突

**位置**：3.3.2 系统歌单 - 最近播放

**问题描述**：
- "分页加载：每页50首，滚动到底自动加载下一页"
- "可配置时间范围（默认7天）：3天、7天、14天、30天、全部"

**核心问题**：
1. **用户选择"全部"时间范围**
   - 假设用户2年内播放过2000首不同的歌曲
   - 分页加载：第1页50首，第2页50首...
   - 但"去重"后实际只有2000首，如何分页？
   - 文档说"每个歌曲只显示最后一次播放时间"

2. **分页加载如何与去重配合？**
   - 首屏加载50首：是最近播放的50首？还是最近50次播放记录（去重后）？
   - 如果是后者，如何确定"最近50次"？
   - 可能需要先查询所有播放记录，去重，再排序，再分页
   - 性能问题

3. **"已加载 X / Y 首"如何计算？**
   - Y是指去重后的总歌曲数？
   - 如果选择"全部"时间范围，Y可能很大（如2000）
   - 如何在不去重全部数据的情况下计算Y？

**影响**：
- 性能问题（查询大量数据）
- 用户体验可能不佳（分页逻辑混乱）

**建议**：
**方案A**（推荐）：时间范围限制
- 移除"全部"选项
- 最大时间范围：365天（1年）
- 这样可以控制数据量，避免性能问题

**方案B**：优化查询逻辑
- 使用数据库索引（lastPlayedDate）
- 查询逻辑：
  1. 先查询符合时间范围的播放记录（分批）
  2. 去重（按songId）
  3. 按lastPlayedDate排序
  4. 分页显示
- 显示"已加载 X 首"（不显示总数Y）

**方案C**：虚拟列表
- 一次性加载数据（在内存中）
- 使用虚拟列表技术（只渲染可见项）
- 适合数据量<1000的场景

#### 【一般】问题10：Song与Playlist的关系模型不够清晰

**位置**：2.2 数据模型

**问题描述**：
```swift
struct Song {
    // ... 没有Playlist相关字段
}

struct Playlist {
    var songIds: [String]  // 包含的歌曲ID列表（有序）
}
```

**核心问题**：
1. **多对多关系的查询性能**
   - 查询某首歌在哪些歌单中：需要遍历所有Playlist
   - 查询某首歌的播放次数在哪个歌单最高：复杂

2. **删除歌曲时的效率**
   - 文档3.1.5："从所有歌单中移除该Song记录"
   - 需要：遍历所有Playlist → 检查songIds → 删除songId
   - 如果有100个歌单，需要100次数据库操作

3. **批量操作的效率**
   - 批量添加100首歌到10个歌单
   - 需要：更新10个Playlist的songIds数组
   - songIds数组可能很大，性能问题

**影响**：
- 数据库查询性能可能不佳
- 大量数据时操作缓慢
- 扩展性差

**建议**：
**方案A**（推荐）：引入关系表
```swift
// 关系表（CoreData会自动生成）
struct PlaylistSong {
    let playlistId: String
    let songId: String
    var sortOrder: Int       // 在歌单中的位置
    let addedAt: Date        // 添加时间
}

// Playlist模型简化
struct Playlist {
    // ... 其他字段
    // 移除 songIds: [String]
}

// Song模型保持不变
```

**优点**：
- 查询效率高（可以建立索引）
- 删除歌曲时：DELETE FROM PlaylistSong WHERE songId = ?（一条SQL）
- 批量操作：批量插入PlaylistSong

**缺点**：
- 数据模型更复杂
- 需要使用CoreData或其他ORM

**方案B**：保持当前设计
- 优化查询：为songIds字段建立索引
- 限制歌单数量：最多100个歌单
- 限制歌单大小：每个歌单最多1000首
- **适用场景**：小规模应用（<10000首歌）

**推荐**：
- v1.0：使用方案B（简化设计）
- v1.1+：如果性能成为瓶颈，迁移到方案A
